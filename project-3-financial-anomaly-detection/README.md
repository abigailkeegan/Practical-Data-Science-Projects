# Project 3: Anomaly Detection in a Financial Dataset

Unsupervised anomaly detection on a financial transactions dataset, comparing a Gaussian Mixture Model (GMM) against an Isolation Forest. Both models are trained on normal transactions only and evaluated against 2,000 synthetic anomalies injected to create ground-truth labels.

**Authors:** Abigail Keegan, Michelle Ng Du

**Course:** CS 667 Practical Data Science, Pace University

## Overview

The dataset contains 216,960 transactions across 15 accounts, 10 merchants, 3 transaction types, and 5 locations, covering January through December 2023. The raw data has only one numeric column (`Amount`), so the project's core work is feature engineering and a careful evaluation setup.

The workflow:

1. **Data preparation:** regex-based timestamp normalization, extraction of temporal features, MinMax scaling of `Amount`
2. **Feature engineering:** 15 features total covering
   * 9 frequency-encoded features (merchant, location, and transaction type usage per account, by month, and as cross-categorical pairs)
   * Scaled transaction amount
   * 5 rolling behavioral features (elapsed time since last transaction, transaction count in last 10 minutes, rolling mean and std of last 20 amounts, location-change flag)
3. **Synthetic anomaly injection:** 2,000 anomalies generated by taking real rows and shifting two features outside the [0, 1] range, giving a ~0.91% contamination rate
4. **Modeling:**
   * **Isolation Forest** tuned with a grid search over number of trees, sample size, contamination, and max features, scored on PR-AUC
   * **GMM** with the number of components selected by BIC and AIC (both agreed on 10), using log-likelihood as the anomaly score
   * Both models trained on normal-only transactions
5. **Evaluation:** precision, recall, F1, ROC-AUC, PR-AUC, confusion matrices, ROC and PR curves, score distributions, PCA projection, and a flagged-anomaly overlap analysis between the two models

## Results

| Model            | Precision | Recall | F1    | ROC-AUC | PR-AUC |
| ---------------- | --------- | ------ | ----- | ------- | ------ |
| Isolation Forest | 0.263     | 0.284  | 0.273 | 0.914   | 0.199  |
| GMM              | 0.913     | 1.000  | 0.955 | 1.000   | 0.930  |

**GMM is the stronger model on this dataset.** The synthetic anomalies sit far outside the bulk of the normal distribution, which translates into very low log-likelihood scores that GMM flags cleanly. Isolation Forest's high ROC-AUC paired with low PR-AUC is the classic rare-anomaly pattern: it ranks anomalies reasonably well on average but cannot pick a threshold that cleanly separates them.

**Overlap between models.** The two models agree on 696 transactions (568 true positives, 128 false positives), but each flags a roughly equal-sized unique group: 1,464 from Isolation Forest only and 1,494 from GMM only. Every Isolation Forest only flag is a false positive, while 96% of GMM only flags are true positives. GMM is doing most of the real work on this dataset, while Isolation Forest is largely picking up unrelated unusual patterns in the rolling features rather than the synthetic anomalies themselves.

**Feature importance for Isolation Forest** is dominated by the rolling time-based features (`elapsed_time`, `rolling_count_10min`, `rolling_std_amt`, `rolling_mean_amt`). Among categorical features, the cross-categorical pairs carry the most signal. *When* and *how often* a transaction occurs is more informative than *which* merchant or category it belongs to.

```bash
pip install -r requirements.txt
jupyter lab notebooks/financial_anomaly_detection.ipynb
```

The notebook loads the dataset with a relative path (`../data/financial_anomaly_data.csv`), so launch Jupyter from this subfolder.

## Dataset

The financial transactions dataset (`financial_anomaly_data.csv`, ~15 MB) is included in this repository. It contains 216,960 transactions with the following fields: `Timestamp`, `TransactionID`, `AccountID`, `Amount`, `Merchant`, `TransactionType`, and `Location`. The dataset has no fraud labels; anomalies for evaluation are generated synthetically inside the notebook.
This is a synthetic dataset provided in the course materials.
